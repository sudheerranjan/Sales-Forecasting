North Star · Guard · Proxy Metric Framework Staff Critical · NEW
Metric TypeDefinitionExample (Video Rec)Failure If Ignored
North Star Metric The single most important long-term business metric. This is what the company ultimately optimizes for. 7-day retention, total watch time per week Optimizing CTR instead leads to clickbait; long-term retention drops despite short-term gains
Guard Metrics Metrics that must NOT degrade, even if north star improves. Defines acceptable trade-off space. Content diversity, p99 latency <200ms, fairness across demographics Model boosts watch time by recommending only viral content; diversity collapses; users churn
Proxy Metric What the model actually optimizes for. Should correlate with north star, but the gap is a trap. CTR, watch completion rate, likes CTR up 15% but 6-month retention drops — clickbait optimized, satisfaction destroyed
Metric Hierarchy Rule: Always define all three levels before choosing an ML objective. Proxy metric must correlate with north star AND must be paired with guard metrics to prevent Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure."
FAANG Probe: "You optimized for CTR and it went up 15%. But 6-month retention dropped. Why?" → CTR is a proxy metric. Maximizing clicks without quality leads to clickbait consumption → long-term churn. North star = long-term retention. CTR needed a quality co-signal (watch completion) as guard metric.

RSMA: 4-Pillar System Requirements Chip Huyen · Staff+
PillarWhat to Say in InterviewSilent Risk If Ignored
Reliability The system must perform correctly even under hardware/software faults. Unlike traditional software, ML fails silently — wrong predictions produce no crash or 404 error. Model returns confidently wrong answers with no alert firing. Revenue drops; no engineer is paged.
Scalability We grow in 3 dimensions: model complexity, traffic volume, and model count. Artifact management (versioning, rollbacks) is as critical as resource auto-scaling. Monolithic model pipeline collapses under load or versioning chaos causes irreproducible models.
Maintainability Code, data, and model artifacts must be versioned. ML engineers, DevOps, and domain SMEs must collaborate — tooling must accommodate all stakeholders. Unreproducible experiments, broken rollbacks, and untracked feature schema changes corrupt models.
Adaptability Data distributions shift continuously. The system must support updates without service interruption and must automatically discover performance regressions before users report them. Model silently degrades as world changes (seasonality, concept drift). No retraining trigger fires.

Capacity Estimation · Deep-Dive Explainers Session Notes · NEW
① QPS → Distributed Inference Fleet

A distributed inference fleet is a pool of servers each running a copy of the model, with a load balancer routing requests across them. Single-node serving becomes the bottleneck once QPS exceeds ~3–5K requests/sec per node.

Video Rec example (1B DAU): QPS = 1B × 5 req/day ÷ 86,400 ≈ 58K QPS Servers = 58K QPS ÷ 3K QPS/server ≈ 20 nodes (+ 2× headroom for spikes) User Request → Load Balancer ↓ ↓ ↓ [Server 1] [Server 2] ... [Server N] Model Model Model copy copy copy ↓ Pre-loaded ANN index shard
Interview move: State the math → declare "this exceeds single-node capacity" → propose the fleet. Don't jump to fleet without the QPS justification.
② Index Size → Faiss IVF / HNSW Sharding

QPS drives how many nodes. Index size drives how those nodes store and query embeddings. These are independent dimensions that combine in serving design.

AlgorithmHow It WorksTrade-off
Faiss IVF (Inverted File)Clusters items into N buckets; searches only nearest buckets at query timeFaster; can miss items at cluster boundaries
HNSW (Hierarchical Navigable Small World)Multi-layer graph; traverses from coarse to fine layersVery fast query (~1ms); high memory overhead
Why sharding? 10B videos × 128-dim × 4 bytes = 5.12 TB → no single node has this RAM Shard across 20 nodes → each holds ~256 GB of the index User Embedding Query ↓ ANN Coordinator (fan-out) ↙ ↓ ↓ ↘ Shard1 Shard2 Shard3 Shard4 ← each holds 1/4 of index ↘ ↓ ↓ ↙ Merge → Top-K returned Item embeddings: batch pre-computed OFFLINE, loaded into shards User embeddings: computed REAL-TIME at query time (user tower)
③ Latency SLA → p99 Sub-Budget Decomposition

The p99 SLA is a total budget. Distribute it across every pipeline hop. The cheatsheet gives explicit targets for Stage 1 (<20ms) and Stage 2 (<100ms). Remaining budget goes to network + user embedding + re-ranking.

Pipeline StageBudget (p99 = 200ms)Source
User Embedding Inference~10–20msUser tower forward pass
ANN Retrieval (Stage 1)<20msCheatsheet explicit target
Heavy Ranker (Stage 2)<100msCheatsheet explicit target
Network RTT (Load Balancer)~20–30ms~10ms per hop, back-of-envelope
Re-ranking (business rules)~10msNo model — pure scoring logic
Key insight: ANN must be <20ms specifically to protect the ranker's budget. If ANN takes 80ms, the ranker only gets 60ms — not enough for a 200-feature DNN over 2K candidates. The constraint flows from p99 inward, not the other way.
Interview move: "My total p99 budget is 200ms. Stage 1 ANN must be under 20ms to leave budget for my Stage 2 ranker which needs ~100ms. Network overhead consumes ~30ms. That leaves ~50ms for user embedding and re-ranking."
④ Compute Budget → Feasibility Trigger, Not a Threshold

The formula retrain_freq × data_rows × model_size does not produce a number you compare against a fixed threshold. Its output is a feasibility question: "Can a full retrain from scratch complete within the retraining window?"

SystemData / CadenceFull Retrain Feasible?Forced Choice
Ad Click500M rows, continuousNo — takes hours; 5-min staleness deadline violatedStateful fine-tuning on delta data → NN only (GBDT eliminated)
Video Rec100M rows, dailyBorderline — feasible with large GPU clusterStateful Stage 3 preferred for efficiency

Decision logic (not a formula output): Step 1: Estimate one full retrain cost → data_rows × model_size → "how big?" Step 2: Compare to retrain window → "does it fit?" Step 3: YES → Stateless feasible → GBDT or NN both valid NO → Must go Stateful (fine-tune on delta) → GBDT eliminated → NN required
Feasibility = retrain duration vs. staleness window (not a fixed threshold)
The staleness window is system-specific and comes from business stakes, not a formula output:
• Ad Click: bid cycle = 5 min → even a 1-hr retrain violates the window → Stateful mandatory
• Video Rec: daily refresh acceptable → 6–8 hr retrain borderline → Stateful preferred
• PYMK: weekly cadence → any reasonable retrain fits → Stateless OK
The staleness window kills feasibility, not raw GPU time alone. Always ask: "How stale can my model be before it costs money or safety?"
Why GBDT gets eliminated: GBDT cannot be incrementally fine-tuned on new data. It must retrain from scratch. For high-frequency retraining (daily/continuous), this is computationally infeasible at FAANG scale. Neural Networks support checkpoint-based stateful training on delta data only.
Interview move: "Daily retrain on 100M rows with a DNN takes ~6–8 hours of GPU time. That exceeds my 24-hour window after validation + deployment. So I go stateful: fine-tune from last checkpoint on delta data only. This eliminates GBDT and locks me into DeepFM or Two-tower NN."
Stakeholder Conflict & Decoupled Objectives Pattern Staff Trap · NEW
Core Rule (Chip Huyen): Never combine conflicting objectives into a single loss function. Optimizing for engagement alone inadvertently promotes extreme/clickbait/harmful content.
Conflict TypeExample❌ Wrong Approach✅ Correct Pattern
Engagement vs. Safety CTR ranker amplifies misinformation Penalize harm inside CTR loss — couples objectives, forces retraining to adjust policy Train separate harm classifier; inject as guard signal at serving time
Revenue vs. User Trust Ad CTR vs. hide-rate conflict Single model optimizing revenue ignores user satisfaction signals Separate CTR model + hide-rate predictor; combine at serving
CTR vs. Quality Clickbait scores high CTR, low completion rate Optimize CTR only; no quality co-signal Final Score = w₁·CTR + w₂·completion_rate − w₃·harm_score
Decoupled Objectives Formula:
Final Score = w₁·engagement + w₂·quality − w₃·harm
Weights are tunable business knobs — no retraining needed to change policy. Train models separately; combine at serving.
FAANG Probe: "You've built a news feed ranker optimized for CTR. T&S says it's amplifying misinformation. Fix it without rebuilding." → Decouple — add a separate harm classifier and quality scorer; re-rank with weighted multi-signal formula at serving. Don't retrain CTR model; add guard rails as new signals.

Combining Loss Functions & Serving Scores — Master Equations Staff Critical · NEW
① Universal Multi-Task Loss (News Feed, Harmful Content, Video Rec)
Combine K independent task losses into one backward pass. Weights are tunable without retraining — this is the business policy knob.
\[ \mathcal{L}_{ ext{total}} = \sum_{k=1}^{K} w_k\,\mathcal{L}_k \quad ext{where} \quad \sum_{k=1}^{K} w_k = 1 \]
SystemKTask HeadsLoss per Head
Harmful Content4–6violence, nudity, hate, self-harm, spamBCE per label (independent, not softmax)
News Feed5like, share, comment, hide, reportBCE per head; w_hide & w_report negative
Video Rec2click (binary), watch-time (float)BCE + Huber; watch-time weighted 5–10×
Trap: Using Softmax across task heads implies mutual exclusivity — a post cannot be both a like and a share. Always use independent BCE per head for multi-label tasks.
② Final Serving / Re-ranking Score — Decoupled Objectives Pattern
Models are trained separately. Scores are combined at serving time — no retraining needed to change policy weights.

Imbalance Loss Function — 3-Question Decision Flow Session Notes NEW
Root cause first. All three techniques fix class imbalance, but attack different root causes. Ask these 3 questions in order before choosing a loss function.
Q1 — Do I know the business cost of FN vs FP?
YES → Cost-Sensitive Learning
Domain expert defines cost matrix CFN vs CFP. Use when FN costs 10×+ more than FP.
e.g., Insurance fraud, Medical diagnosis, Street View (FN = legal liability)
NO → continue ↓
Q2 — Are there millions of obviously-correct negatives drowning the gradient?
"Easy negatives" = examples the model already predicts correctly with high confidence (e.g., 99.9% non-clicks, cat photos in harm detection, clear non-fraud transactions)
YES → Focal Loss (γ=2 default)
L = −(1−p)γ log(p). Mathematically silences easy examples. Model capacity goes entirely to hard, ambiguous cases.
e.g., Ad Click (0.1% CTR), Harmful Content (obvious safe posts dominate), Object Detection
NO → continue ↓
Q3 — Is it purely a frequency problem — too few labeled positives?
YES → Class-Balanced Loss
Weight = nmajority / nminority. Makes model treat 50K self-harm examples as if they were 5M.
e.g., Multi-task harmful content (self-harm 50K vs violence 5M), any rare-event classifier as baseline
Production Combination Pattern — Stack in order of specificity:
Focal Loss (silences easy negatives per example) + Class-Balanced (fixes frequency gap across tasks in multi-task models) + Cost-Sensitive threshold at serving time (business policy, no retrain needed).
⚠ Never resample large transformer-scale models — resampling breaks calibration; P(y|x) no longer matches the real-world class prior. Use two-phase training instead.

Data Leakage — 4 Silent Sources That Kill Production Models Chip Huyen NEW
Data leakage = future information enters training data, inflating offline metrics but causing silent failure in production. The model learns shortcuts that don't exist at serving time.
Leakage SourceHow It HappensFix
1. Scaling before splitting Computing mean/variance on the full dataset (train + test) leaks test statistics into training Always split first, then fit the scaler exclusively on the train split. Apply to test.
2. Imputing before splitting Filling missing values using median/mean computed on full data — test distribution leaks into train Split first, then compute imputation statistics on train only. Transform test with train stats.
3. Temporal leakage Random shuffling time-series data — model trains on events after the prediction target Always use temporal splits: train on past, validate/test on future. Never random-shuffle across time.
4. Process leakage A feature correlates with the label not through causality but because it was generated after the label event (e.g., scan machine tag assigned post-diagnosis) Audit feature creation timestamps. Ensure every feature in training was available before the label event.
Leakage Detection Checklist:
  • Suspiciously high offline metrics (near-perfect AUC)? → First check for leakage before celebrating
  • Feature importance dominated by one unexpected feature? → Check its creation timestamp vs. label event
  • Offline metrics collapse when you switch to a strict temporal split? → You had temporal leakage
  • New items/users score perfectly at launch but degrade fast? → Process leakage or future-dependent feature
FAANG Probe: "Your fraud model has 99.5% AUC offline but barely beats random in A/B test. What happened?" → Classic leakage. Check: (1) Was data scaled/imputed before splitting? (2) Was there random shuffling of time-ordered fraud events? (3) Does any feature encode post-fraud-event metadata (e.g., account status updated after fraud was confirmed)?

Feature Crossing — When to Do It and Why Chip Huyen NEW
Feature crossing creates new features from combinations of existing ones (e.g., marital_status × num_children). Whether you need it depends entirely on your model family — candidates who apply it blindly signal shallow knowledge.
Model FamilyNeed Feature Crossing?Why
Logistic Regression / Linear models ✅ Essential Linear models cannot learn non-linear interactions. Without crossing, marital_status=married AND num_children=3 is invisible to the model. You must construct it manually.
GBDT / XGBoost Sometimes helpful Trees naturally split on feature combinations — but explicit crossing on high-signal pairs can speed up convergence and reduce tree depth needed.
Neural Networks (DNN, DeepFM) Less critical — but explicit crossing still helps DNNs learn interactions implicitly. But DeepFM/DCN explicit crossing accelerates learning of low-order interactions and improves sample efficiency.
Caveat to always raise: Feature crossing blows up the feature space — 100-cardinality × 100-cardinality = 10,000 combinations. This risks overfitting, memory blowup, and training instability. Use embedding-based crossing (FM, DeepFM) to handle high-cardinality crosses efficiently.
FAANG Probe: "Would you use feature crossing for your news feed ranker?" → For the DNN backbone: crossing is handled implicitly and by the FM component of DeepFM. For ablation: test whether explicit crosses on user_age × content_category improve nDCG. Raise the cardinality risk proactively.

Encoding brand/category/item as a fixed integer ID silently breaks in production when new categories appear continuously. This is the most common categorical feature production bug.
ApproachProblem in ProductionFix
Integer ID encoding (brand=3, brand=47...) New brand at inference → out-of-vocabulary → undefined behavior or crash. Model learned nothing about unseen categories. Never use raw IDs for open-vocabulary categoricals.
One-hot (top-N + UNKNOWN bucket) Acceptable if N is stable. But model must be trained on UNKNOWN — if it never sees UNKNOWN during training, it learns nothing for OOV categories. Ensure training data includes UNKNOWN examples. Reserve 5–10% of training for UNKNOWN bucket.
Embedding layer ✅ New categories get their own embedding initialized randomly — but with a warm-start from similar seen categories via fallback. Best for high-cardinality open-vocabulary categoricals. Generalizes via embedding similarity. Required for cold-start new items.
Hashing trick (feature hashing) Maps any string to a fixed-size bucket — handles new categories automatically with no vocab update needed. Hash collisions possible. Use for very high cardinality (hashtags, user IDs) where collisions are tolerable.
Production Rule: For any categorical feature where new values appear post-deployment (brands, advertisers, event hosts, content creators), always use embeddings or hashing. One-hot with fixed vocab requires a model redeploy every time a new category appears.
Ablation Studies — Feature Importance at Staff Level Chip Huyen NEW
An ablation study systematically removes components (features, modules, loss terms) one at a time and measures the performance drop. It is the rigorous way to validate that your feature engineering choices actually help — and to prune dead weight before serving.
FindingWhat It Tells YouAction
Removing feature X causes significant metric drop Feature X is load-bearing. Model depends on it. Invest in feature quality, freshness, and monitoring for X. Add drift alert for X's distribution.
Removing feature X causes no change in metric Feature X is dead weight — redundant, noisy, or already captured by another feature. Drop it. Reduces serving latency, memory, and potential leakage risk.
Removing feature X improves metric Feature X is causing harm — possibly leakage, spurious correlation, or adversarial noise. Investigate immediately. Check creation timestamp vs. label event. Check for leakage source.
Staff Badge: After proposing your feature set, say: "Post-training, I'd run ablation studies to validate which features actually contribute. Features that don't improve performance add serving cost and can introduce leakage risk — I'd prune them aggressively." This signals production-first thinking, not just accuracy-chasing.

CLASSIFICATION OUTPUTS
Binary click/no-click, spam/not {0, 1} Binary Cross-Entropy (BCE) Standard. Use sigmoid output layer. Calibrated probabilities. Using MSE → gradients vanish near 0/1, model trains poorly
Multi-class fixed N classes {0…N-1} Cross-Entropy + Softmax Output is probability distribution over N classes. Only valid when N is fixed and small (<10K). Using for dynamic catalogs (500M items) → softmax infeasible; use BCE with sampled negatives instead
Multi-label harm types, tags [0,1]^N independent BCE per label, summed Each label is independent binary. Sum BCE across all label heads. Used in harmful content multi-task. Using softmax forces mutual exclusivity — labels compete instead of being independent
Imbalanced binary fraud, rare harm {0, 1} skewed Focal Loss Down-weights easy negatives, forces model to focus on hard positives. γ=2 standard. Use when positive rate <1%. Using plain BCE → model predicts all negatives, gets 99% accuracy, zero recall on positives
REGRESSION OUTPUTS — CONTINUOUS
Unbounded float, symmetric house price, ETA (-∞, +∞) MSE Penalises large errors quadratically. Best when large errors matter more than small ones. Sensitive to outliers — a few bad labels dominate training
Unbounded float, outlier-robust watch time, session length (-∞, +∞) Huber Loss MSE for small errors, MAE for large errors. Threshold δ is tunable. Best default for production regression. MAE alone gives subgradients at 0 — unstable in DNNs without careful tuning
Bounded continuous score in [0,1], probability [0, 1] BCE or MSE (both valid) If output is a calibrated probability → BCE. If output is a similarity score → MSE. Use sigmoid output layer. Predicting probabilities with MSE doesn't constrain outputs to [0,1] — add sigmoid or clip
Positive skewed float revenue, ad spend (0, +∞) skewed right Log-space MSE Apply log(y+1) transform before MSE. Converts multiplicative errors to additive. Predicts order-of-magnitude correctly. MSE on raw revenue → high-value outliers dominate; model underpredicts median cases
Count / rate data ride demand, QPS, requests/min {0,1,2…} or rate ≥ 0 Poisson Loss Variance grows with mean — errors at low counts are proportionally catastrophic. Formula: ŷ − y·log(ŷ). Use softplus output. MAE/MSE treats error of 10 at prediction 12 same as at prediction 1200 — silent underfitting at low-demand cells
RANKING / ORDERING OUTPUTS
Pointwise rank CTR score per item Score per item BCE (binary label) or Huber (continuous) Simplest. Predict relevance score independently per item, sort descending. Ignores inter-item ordering. Position bias — clicks at position 1 ≠ relevance. Debias with IPS before treating click as ground truth.
Pairwise rank A preferred over B Pair preference BPR / RankNet loss Optimises relative ordering between pairs. Requires editorial labels or explicit A>B signals. Better ranking quality than pointwise. Needs graded labels (0–3) or explicit reordering — click-only data is insufficient
Listwise rank full ordering, search Full list order LambdaRank / ListNet Directly optimises nDCG. Most complex, best quality. Use only when rich editorial label data available. Without rich labels, listwise degrades — underfits on click-only data
REPRESENTATION / EMBEDDING OUTPUTS
Contrastive embeddings visual search, two-tower Vector in R^d InfoNCE / NT-Xent Pulls positive pairs together, pushes negatives apart in embedding space. Temperature τ controls sharpness. Used in SimCLR, CLIP. Using MSE on embeddings → no negative pushing, embeddings collapse to zero
Negative sampling W2V-style, Airbnb listings Score per pair CE over pos + sampled negatives Efficient approximation of full softmax. Sample k random negatives per positive. Hard negatives (in-batch) improve quality. Random negatives only → easy negatives dominate; add hard negative mining for dense retrieval quality

Poisson Loss — Regression on Count Data Session Note NEW
Common Trap: Using MAE or MSE for count/rate outputs (ride demand, QPS, event requests). Both treat a prediction error of −10 identically whether you're predicting 12 or 1200. That's wrong — variance in count data grows with the mean.
When to Use Poisson Loss
SignalUseWhy
Output is a non-negative integer count (ride requests, pageviews)Poisson lossVariance scales with mean — errors at low counts are proportionally far more serious
Output is a continuous rate but skewed (revenue, duration)Log-space MSETake log(y+1) then apply MSE — equivalent approximation, simpler to implement
Output is unbounded float, roughly Gaussian (house price)MSE / HuberStandard regression — symmetric errors, no count structure
Poisson Loss Formula
Loss(ŷ, y) = ŷ − y · log(ŷ) Where: ŷ = predicted count (must be > 0, use softplus activation) y = actual observed count Intuition: penalises relative errors — missing 10 rides on a cell expecting 15 is catastrophic; same miss on 500 is noise. Practical tip: if using PyTorch/TF, use PoissonNLLLoss or train in log-space → output log(ŷ), use MSELoss on log targets.
FAANG Probe: "Your surge pricing model predicts ride request rates. Why not use MAE?" Ride requests are count data — variance grows with the mean. MAE treats an error of 10 at a predicted value of 12 the same as an error of 10 at 1200. Poisson loss weights errors proportionally, penalising relative misses at low-demand cells more heavily. I'd use Poisson loss or log-space MSE as a practical alternative.

Ranking Metrics — Formula Reference NEW
PRECISION@K
\(\displaystyle\text{Precision@}K = \dfrac{|\{\text{relevant items in top-}K\}|}{K}\)
K = cut-off rank  ·  numerator = count of relevant items in top-K positions
ⓘ Does not penalise rank position within the window — quick sanity check only.
RECALL@K
\(\displaystyle\text{Recall@}K = \dfrac{|\{\text{relevant items in top-}K\}|}{|\{\text{all relevant items in catalog}\}|}\)
denominator = total relevant items in full catalog (not just retrieved)
ⓘ Primary metric for Stage 1 ANN retrieval. Low Recall@K = good items invisible to ranker.
AVERAGE PRECISION (AP) & MAP
Step 1 — AP for one query
\(\displaystyle\text{AP} = \frac{1}{|R|} \sum_{k=1}^{K} \text{Precision@}k \;\cdot\; \mathbb{1}[\text{item at rank }k\text{ is relevant}]\)
Step 2 — Mean over all queries
\(\displaystyle\text{MAP} = \frac{1}{Q}\sum_{q=1}^{Q}\text{AP}_q\)
|R| = total relevant in catalog  ·  𝟙[·] = indicator (1 if relevant, else 0)  ·  Q = number of queries
ⓘ Rewards early hits. Divides by |R| (not just retrieved) — penalises missed relevant items. Used: Event Rec, PYMK, Street View (mAP@IoU).
MEAN RECIPROCAL RANK (MRR)
\(\displaystyle\text{MRR} = \frac{1}{Q}\sum_{q=1}^{Q}\frac{1}{\text{rank}_q}\)
rankq = position of the first relevant item for query q  ·  Q = number of queries
ⓘ Only cares about the first hit — use only when exactly one canonical correct answer exists. Used: YouTube Video Search.
nDCG — NORMALISED DISCOUNTED CUMULATIVE GAIN
Step 1 — DCG@K
\(\displaystyle\text{DCG@}K = \sum_{k=1}^{K}\frac{2^{\,\text{rel}_k}-1}{\log_2(k+1)}\)
Step 2 — Ideal DCG (perfect ranking)
\(\displaystyle\text{IDCG@}K = \sum_{k=1}^{K}\frac{2^{\,\text{rel}^*_k}-1}{\log_2(k+1)}\)
Step 3 — Normalise
\(\displaystyle\text{nDCG@}K = \frac{\text{DCG@}K}{\text{IDCG@}K}\)
relk = graded relevance at rank k (e.g. 0–5)  ·  rel*k = ideal grade at rank k  ·  log₂(k+1) = position discount
ⓘ Use when relevance is graded (0–5), not binary. Subsumes MAP for binary labels. Used: Visual Search, Video Rec, News Feed.
Metric selection rule — state these two facts before naming any metric:
1. Relevance type: binary (click / no-click) → MAP or MRR  ·  graded (0–5 editorial) → nDCG
2. Answer count per query: one correct answer → MRR  ·  multiple relevant items → MAP / nDCG

“How do you evaluate a new ranking model before A/B testing?”

“You said optimise for CTR — how do you handle clickbait?”

⑩ Debugging Production Inference Failures
The D.I.A.L. Framework · Chip Huyen Ch.8 & Ch.10 · Silent failure detection → root cause attribution
⚠ The Core Problem: ML Fails Silently
Chip Huyen's #1 insight: ML systems fail silently — the code compiles, the endpoint returns 200, the loss looked fine in training, but predictions are quietly wrong. A traditional software failure gives you a 404 or segfault; an ML failure gives you confident wrong answers with no alert firing.
TRADITIONAL SOFTWARE FAILURE
  • Returns 404 / 500
  • Throws a segfault
  • Empty response body
  • Timeout / OOM
  • ✅ Alert fires immediately
ML SYSTEM FAILURE (SILENT)
  • Returns HTTP 200
  • Loss was fine in training
  • Predictions shift quietly
  • Revenue drops, no page
  • ❌ No alert fires without monitoring

Layer D — Detect: Is There a Real Failure?

Two types of expectation violations — distinguish them before acting:

TypeExamplesHow It Surfaces
Operational (easy) Timeout, OOM, 404, segfault, empty response Pages you immediately
ML Performance (silent) Accuracy drifts, predictions shift, proxy metrics decouple from north star Requires monitoring to surface
DETECTION CHECKLIST
  • p99 latency spiking? → Operational, not model quality
  • Prediction distribution shifting (more zeros, extreme scores)? → ML signal
  • User feedback (hide rate, completion rate, downvotes) degrading while CTR holds? → ML objective misalignment
  • Accuracy on held-out slices dropped? → Distribution shift
🔴 FAANG Probe: Your model has 95% accuracy offline but revenue dropped post-deployment. What happened?
Offline metric diverged from business KPI. Likely distribution shift, or ML objective (CTR) misaligned with north star (revenue). Fix: add online A/B metrics tied to business KPIs.
Layer I — Isolate: Software Bug or ML Failure?
Google data (Chip Huyen Ch.8): 60 of 96 ML pipeline breaks over 15 years were non-ML causes — distributed systems errors, orchestration mistakes, incorrect data joins. Always rule out software first.
SOFTWARE FAILURE CHECKLIST (RULE OUT FIRST)
  • Wrong model binary deployed — did you push the right version? Check model registry artifact
  • Dependency version change — did a library upgrade silently change behavior?
  • Permission/read error — is the serving system reading the correct feature source?
  • Orchestration failure — did a DAG step fail silently, causing stale features to be served?
  • Hardware fault — GPU/CPU behaving incorrectly under thermal load
If all software causes are ruled out → proceed to Layer A (ML-specific).
Layer A — Attribute: The 4 ML Root Causes (Chip Huyen Ch.8)
A1. TRAIN-SERVING SKEW (Most Common)

Model is correct, feature list is correct, but featurization code at serving computes features differently from training.

Skew TypeCauseFix
Feature skewTraining batch features ≠ serving compute (Python→Java rewrite)Feature Store (Feast/Tecton) shared transform code
Label skewTraining labels from period no longer representativeTemporal train/eval splits + drift monitoring
Log-and-WaitFeatures logged at prediction time ≠ features recomputed at training timeLog features from prediction service; reuse at training
🔴 Probe: Model works offline, degrades first week in prod → Train-serving skew. Fix with Feature Store.
A2. DATA DISTRIBUTION SHIFT

The world changed; the model didn't. Three subtypes:

Shift TypeDefinitionDetection
Covariate shiftP(X) changes, P(Y|X) stays samePSI / KS test on features
Label shiftP(Y) changesMonitor prediction dist. over time
Concept driftP(Y|X) itself changesAccuracy degrades even with stable input/output dist.
Detection tools: KS test, PSI (PSI > 0.2 = major shift → retrain). Monitor prediction distribution shift as a cheap proxy for input shift.
A3. MODEL STALENESS / DEGENERATE FEEDBACK LOOP

Model was fine at launch but degraded as the world evolved. A degenerate feedback loop is a special case: the model's own predictions influence future training data, amplifying its biases.

Example: Recommender shows cat videos → cat lovers engage → system labels cats as "good" → recommends more cats → attracts more cat lovers → loop locked in.

Fix: Monitor diversity metrics as guard metrics; inject exploration (ε-greedy); use counterfactual/IPS-debiased training data.

A4. EDGE CASES

Outlier inputs the model was never trained on. A single edge case can reveal fragility in the entire pipeline.

  • Example: Never-seen user segment, new device type, unusual text encoding
  • Detection: Slice-based monitoring — track performance per subgroup, not just aggregate

Monitoring Toolbox for Debuggability (Observability)
Chip Huyen's definition: Observability = the ability to infer internal system state from runtime outputs without shipping new code.
📋 LOGS
  • Record every event with unique trace IDs, timestamps, user IDs, input hashes
  • Use distributed tracing — a 20–30 hop request must be fully traceable
  • Tag logs with subgroup metadata (zip code, device, segment)
📊 DASHBOARDS
  • Visualize feature distributions and prediction distributions
  • Latency percentiles — always track p50 AND p99, never just average
  • Slice-based accuracy by user segment, geography, device
🚨 ALERTS
  • PSI > 0.2 on key features → major distribution shift
  • Prediction distribution shift beyond threshold
  • p99 latency SLA breach
  • Avoid alert fatigue — only alert on actionable, critical signals
